# Rebio

**仕事は情報ではなく、責任が流れている。**

> Rebioは、記録・申し送り・タスク・書類・コメントをつなげることで、業務上の責任が誰に渡り、どこで止まり、どう解決されたかを見える化する業務共有SaaSです（実用版を見据えたデモMVP）。

単なる申し送りアプリでも、ファイル管理アプリでも、タスク管理アプリでも、チャットアプリでもありません。「誰がボールを持っているか」「どこで止まっているか」「何を根拠に対応したか」「最後にどう解決したか」を追えるようにすることを目指しています。

## 開発背景

医療・介護・警備・小売・物流など、シフト制で動く現場では、口頭や紙ベースの申し送りが今も多く残っています。

現場で感じた課題：

- 重要な情報が次の担当者に届かず、対応が遅れる
- 「記録した」だけで終わり、タスクとして引き継がれない
- マニュアルや点検表が散在していて、申し送りとつながっていない
- 担当者が変わるたびに「どこに何が書いてあるか」がわからなくなる
- 誰がボールを持っているか、どこで止まっているかが見えない

これらを「構造的に」解決するために、Rebioを開発しました。

## 設計思想：普段は迷わない、必要な時だけ深く使える

Rebioは「多機能に見せること」より「毎日の業務で迷わず使えること」を優先しています。

| 原則 | 実装 |
|---|---|
| **入力はシンプル** | テキストの自由入力を基本とし、クイック入力テンプレートで補助 |
| **足りない情報だけ確認する** | あいまいな入力（担当者未設定・報告必要かも）にだけヒントを表示 |
| **AIは裏側で働く** | 自動整理・タスク生成・タグ付けは目立たずに行い、結果だけ提示 |
| **重要なものだけ目立たせる** | 「対応が必要」な申し送りだけ色・アイコンで前面に、共有事項は軽量表示 |
| **タスクを強制しない** | タスク化は「対応が必要」な申し送りだけ。共有事項や通常記録はタスクにしない |
| **ナビゲーションは4本に絞る** | ホーム・記録・申し送り・書類。タスクは申し送りから自然に見える |
| **詳細設定は奥に隠す** | フィルター・ステータス変更・修正は展開してから表示 |
| **書類とタスクは自然につながる（強制はしない）** | 申し送りに関連書類を表示できるが、単なるファイル置き場としても使える |
| **使用者が用途を決められる余白を残す** | カテゴリ・タグ・確認レベルは自由設定。アプリ側で業務フローを決めすぎない |

現場の担当者が「次に何をすべきか」を迷わず判断できる画面を目指しています。

## Rebioはチャットアプリではない

v11では送信対象指定・複数人の既読確認・コメント機能を追加しましたが、Rebioの軸は変えていません。

- **申し送りは全体だけでなく、部署・個人・役職・シフトに向けて共有できる**ようにしました。ただし対象を選ぶUIは軽量なチップ選択にとどめ、権限管理や組織図編集のような重い機能は持たせていません
- **コメントはチャットではなく、申し送りへの補足として設計しています**。返信・スレッド・リアクション・メンションは実装せず、1件の申し送りに対する時系列の追記に限定しています
- **Rebioは情報を必要な相手に届ける業務共有SaaSであり、チャットアプリではありません**。個人チャットやDM、通知センターのような機能は意図的にスコープ外としています

## デザイン方針

Rebioは、現場の人が落ち着いて使えるよう、強い色や過剰な装飾を避け、必要な情報だけが自然に目に入るクリーンなUIを目指しています。

- **彩度を抑えたカラーパレット** — ブランドカラーは落ち着いたブルー・ティール系。緊急・重要な警告以外では強い色を使わない
- **薄いグレー背景に白カード** — 影は最小限にとどめ、境界線（border）で構造を示す
- **申し送りの重みに応じた表示** — 「共有事項」は軽く、「対応が必要」だけ少し強く。全体が警告色だらけにならないようにする
- **角丸10〜14px程度で統一** — バッジはピル型、カードはやや丸みのある四角

## 解決したい課題

| 課題 | Rebioのアプローチ |
|---|---|
| 申し送りの抜け漏れ | AIが候補を整理・カテゴリ分類し、確認フローで見落としを防ぐ |
| 記録がタスクに転化されない | タスク候補を確認した申し送りからタスクを自動生成 |
| 重要情報の埋もれ | 重要度・信頼度の明示、確認必須フラグで目立たせる |
| 書類が散在する | 業務書類を申し送り・タスクと紐づけてLibraryで一元管理 |
| 現場が異なると使えない | カテゴリ・タグを自由設定し、業種に依存しない設計 |

## 主な機能

### 1. 勤務ログ入力（`/logs/new`）
- 勤務種別・本文・次の担当者・重要度を記録
- 業種別サンプルログ・クイック入力テンプレートで即体験できる
- 入力後、Rebioが申し送り候補を自動整理

### 2. 整理候補の確認画面（`/handover/review`）
- 処理フロー（ログ解析→候補抽出→重要度判定→確認必須判定→本人の確認）を可視化
- 「共有事項」「確認してほしい」「対応が必要」の3種類に自動分類
- 重要度・信頼度・元ログの対応箇所を全候補に表示
- 「確認済み」「修正する」「不要」の3択で一件ずつ判断

### 3. 申し送りを確認（`/handover`）
- 申し送りは必ずしも対応タスクではない、という考え方を反映
  - **共有事項** … 対応不要。軽量なカードで表示
  - **確認してほしい** … 既読・確認状態を表示（既読確認なし／既読のみ／確認必須から選択）
  - **対応が必要** … タスク化・期限・担当者・ステータスを前面に表示
- カテゴリ・重要度・ステータス・担当者でフィルタリング、キーワード検索
- **送信対象指定（v11）** … 全体・部署・個人・役職・シフトの5種類から、誰に向けた共有かを指定できる。複数対象の指定も可能（例：夜勤チーム＋管理者）
- **既読・確認の集計（v11）** … 対象が複数人の場合、「既読 2/4」「確認 1/3」のように件数で状況を把握できる。未読者名を前面に出す設計はあえて避けている
- **申し送りへの補足コメント（v11）** … 「確認しました」「明日対応します」のような定型文または自由入力で、対応の経過を申し送りに残せる。チャットではなく申し送りに従属する補足欄として設計

### 4. 次の行動（`/tasks`）
- 申し送りのうち「対応が必要」と判断されたものだけを自動生成（共有事項・通常記録はタスク化しない）
- 期限・担当者・完了管理
- 期限超過を赤色で視覚的に警告

### 5. 勤務ログ一覧（`/logs`）
- 過去のログを勤務種別でフィルタリング
- 生成された申し送り件数の表示

### 6. 書類（`/library`・`/library/[id]`）
- 左：カテゴリ一覧／中央：ファイル一覧／右：選択中ファイルの簡易プレビュー、というエクスプローラー型UI
- PDF・Excel・Word・PowerPoint・CSV・画像のプレビュー（モック）
- カテゴリ・タグを自由設定し、アプリ側で用途を決めすぎない設計
- ピン留め・最近開いた書類、申し送り・タスクとの紐づけ（任意）
- リスト表示／コンパクト表示の切り替え

### 7. ダッシュボード（`/`）・デモページ（`/demo`）
- 対応が必要な件数・緊急件数・未完了タスク・本日ログ数のサマリー
- デモ操作ガイド付きの導線カード
- デモページでRebioの概要・設計意図・画面案内を確認可能

## AIを使う上での安全設計

Rebioの最重要設計方針は「AIに任せきりにしない」ことです。

| 安全設計 | 実装内容 |
|---|---|
| **AIは候補を提示するだけ** | 生成された申し送りは「整理候補」として扱い、登録前に必ず本人が確認 |
| **確認必須フラグ** | 「対応が必要」な項目と、緊急・高・低信頼度の「確認してほしい」項目は確認なしでは登録不可 |
| **元ログの根拠表示** | 全候補に「元ログのどの部分から抽出したか」を表示 |
| **確認時間の警告** | ページ開封から30秒未満で登録しようとすると警告モーダルを表示 |
| **断定的な表現の回避** | 「AIが判断しました」ではなく「候補を作成しました」「確認してください」に統一 |
| **処理フロー可視化** | 整理のステップ（解析→抽出→判定→確認）を画面上部に明示 |

## 書類機能の設計意図

Rebioの書類機能は、アプリ側で用途を決めすぎない「シンプルなファイル管理」を意図しています。

- **用途を決めすぎない** — 「点検表専用」「マニュアル専用」といった機能は用意せず、カテゴリ・タグは自由設定
- **必要なら紐づけられる（必須ではない）** — 書類は申し送り・タスクと紐づけできるが、単なるファイル置き場としても使える
- **プレビューは主張しすぎない** — 一覧の右側に選択中ファイルの簡易プレビューを表示する程度にとどめる
- **複数形式のプレビュー** — PDF・Excel・Word・PowerPoint・CSV・画像に対応（デモ用モック実装）。アップロード・ダウンロードは今後の拡張予定

## 起動手順

### localStorageデモモードで起動する（デフォルト）

```bash
# 依存関係をインストール
npm install

# 開発サーバーを起動
npm run dev

# ブラウザでアクセス
open http://localhost:3000
```

環境変数を何も設定しなくても、自動的にlocalStorageデモモードで動作します。

### Supabaseモードで起動する

```bash
# 1. .env.local.example を .env.local にコピー
cp .env.local.example .env.local

# 2. .env.local を編集し、Supabaseの接続情報を設定
# NEXT_PUBLIC_REBIO_DATA_MODE=supabase
# NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
# NEXT_PUBLIC_SUPABASE_ANON_KEY=xxxxx

# 3. Supabase側でテーブルとStorage bucketを作成
#    docs/supabase-schema.sql を参考にテーブルを作成
#    Storage > New bucket で "documents" という名前のバケットを作成

# 4. 開発サーバーを起動
npm run dev
```

接続情報が正しくない場合や未設定の場合は、自動的にlocalStorageデモモードにフォールバックします。

### ビルド確認

```bash
npm run build
```

### Vercelへのデプロイ

1. GitHubリポジトリと連携し、Vercelで新規プロジェクトを作成
2. Framework Presetは自動的に「Next.js」が選択される
3. Supabaseモードで運用する場合は、Vercelの Project Settings > Environment Variables に以下を設定
   - `NEXT_PUBLIC_REBIO_DATA_MODE` = `supabase`
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
4. localStorageデモモードのまま公開する場合は環境変数の設定は不要

## 権限の見せ方

本格的な権限管理画面はまだ実装していませんが、将来的に以下の4段階を想定しています。

| 権限 | 想定できること |
|---|---|
| **管理者** | チーム全体の申し送り・タスク・書類の閲覧と管理、部署・役職・シフトの設定、メンバーの追加・権限変更 |
| **リーダー** | 自部署の申し送り・タスクの閲覧と対応、メンバーへの申し送り送信 |
| **一般メンバー** | 自分に関係する申し送り・タスクの閲覧・対応・コメント、勤務ログの入力 |
| **閲覧のみ** | 申し送り・書類の閲覧のみ。ステータス変更やコメント投稿はできない |

現時点ではすべての操作が一般メンバー相当の権限で行える単一ユーザーのデモです。`docs/supabase-rls.md` に、実装時のアクセス制御方針を整理しています。

## デモデータについて

初回アクセス時に以下のサンプルデータが自動投入されます。

- **勤務ログ** 8件（介護・警備・小売・物流の4業種）
- **申し送り** 12件（カテゴリ・重要度・信頼度付き）
- **タスク** 6件（対応中・未着手・完了を含む）
- **書類** 9件（PDF・Excel・Word・PowerPoint・CSV・画像）

**すべて架空のデータです。** 実在する個人・企業・施設のデータは一切含まれていません。

データはlocalStorageに保存されるため、ブラウザを変えると初期化されます。  
データをリセットしたい場合は、ブラウザのDevTools > Application > LocalStorageからクリアしてください。

## Supabase移行設計と実装状況（v12〜v13）

現在はlocalStorageベースのデモMVPですが、Supabaseへの移行設計と、切り替え可能な実装の土台を追加しています。

### 設計（v12）
- **データベース設計**：`docs/supabase-schema.sql` にテーブル定義を用意（profiles・teams・handovers・tasks・documentsなど15テーブル）
- **RLS方針**：`docs/supabase-rls.md` に、チーム境界・送信対象に基づくアクセス制御方針を整理
- **認証方針**：Supabase Authを使い、メールログインまたはGoogleログインを想定。ログイン後に `profiles` を作成し、`team_members` で所属チームを管理する
- **移行ロードマップ**：`docs/production-roadmap.md` に、v13（データ永続化）・v14（ファイルストレージ）・v15（運用仕上げ）の計画を記載

### データモードの切り替え（v13）

Rebioは環境変数 `NEXT_PUBLIC_REBIO_DATA_MODE` で動作モードを切り替えられます。

| 値 | 説明 |
|---|---|
| `local`（デフォルト） | localStorageのみで動作するデモモード。環境変数が未設定の場合も自動的にこちらになる |
| `supabase` | Supabaseに接続して動作するモード。`NEXT_PUBLIC_SUPABASE_URL` と `NEXT_PUBLIC_SUPABASE_ANON_KEY` の設定が必要 |

`supabase` を指定していても接続情報が欠けている場合は、自動的に `local` モードにフォールバックします。アプリがクラッシュすることはありません。

### 環境変数

`.env.local.example` をコピーして `.env.local` を作成してください。

```env
NEXT_PUBLIC_REBIO_DATA_MODE=local
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
```

### repository層

`app/lib/repositories/` に、データ取得・保存をモードに応じて切り替えるrepository層を新設しました（`handovers.ts` / `workLogs.ts` / `tasks.ts`）。v13時点ではSupabaseへの読み書きロジックは一部未実装で、Supabaseモードでも実際のデータ操作はlocalStorageに委譲されます（次フェーズで本実装予定）。既存ページは引き続き `app/lib/store.ts` を直接利用しており、動作に影響はありません。

### ログイン画面

`/login` ページを追加しました。localモードではログイン不要で「デモユーザーとして続ける」ボタンからそのまま利用できます。Supabaseモードではメールアドレス入力フォームを表示します（実際の送信処理は次フェーズで実装）。

**この構成はまだ実運用を保証するものではありません。** 将来的に認証・チーム管理・クラウド保存・ファイル管理へ拡張できる構成を明文化し、土台を作った段階です。

## 実ファイルアップロード（v14）

Supabaseモードでは、`/library` から実際のファイルをアップロードできます。

- **Storage bucket**：`documents`
- **対応ファイル**：PDF・PNG・JPG・JPEG・WEBP・CSV（上限10MB）
- **保存内容**：Supabase Storageに実ファイル本体を保存し、`documents` テーブルにメタデータ（ファイル名・カテゴリ・タグ・説明・サイズ・アップロード者）を保存
- **localStorageデモモードでは実アップロードはできません**。アップロードを試みると「デモモードのため利用できません」という案内が表示されます

### エラー処理

以下の状況では、原因に応じた案内を表示します。

- Supabase未設定（localモード）でアップロードを試みた場合
- 非対応のファイル形式を選んだ場合
- ファイルサイズが10MBを超える場合
- Storage bucketが未作成の場合
- 通信エラーが発生した場合

### 削除確認

書類の削除は、Supabaseモード・localモードいずれでも確認ダイアログを表示してから実行します。誤操作による削除を防ぐため、確認なしでの削除はできません。

## 実運用データを扱わないことについて

**本アプリはポートフォリオ・デモ用途として開発されています。**

実運用に向けて以下の追加実装が必要です：

- 認証・認可（Auth.js・Supabase Auth 等）
- サーバーサイドのデータ永続化（PostgreSQL・Supabase 等）
- 監査ログ・データ暗号化
- 個人情報保護方針・利用規約の整備
