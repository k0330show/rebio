# Rebio

**仕事の流れと責任の所在を見える化する業務共有アプリ**

> Rebioは、記録・申し送り・タスク・書類・コメントをつなげることで、業務上の情報が誰に渡り、どこで止まり、どのように対応されたかを確認できる業務共有SaaSのデモMVPです。

申し送りやタスク、業務書類はそれぞれ別々に管理されがちですが、実際の現場ではそれらがつながって動いています。  
Rebioでは、「誰が確認したか」「誰が対応するか」「何を根拠に判断したか」「最後にどう解決したか」を追いやすくすることを目指しています。

## 公開デモ

デモURL: https://rebio-lac.vercel.app/

※ 現在はlocalStorageベースのデモMVPです。実運用版ではなく、認証・チーム管理・クラウドDB保存・権限管理などは今後の拡張想定です。

## 開発背景

医療・介護・警備・小売・物流など、シフト制で動く現場では、口頭・紙・チャット・ファイル共有など、複数の方法で情報がやり取りされています。

その中で、次のような課題が起こりやすいと感じました。

- 重要な情報が次の担当者に正しく伝わらない
- 記録は残っていても、誰が対応するのかが曖昧になる
- マニュアルや点検表などの書類が、申し送りやタスクとつながっていない
- 担当者が変わると、過去の経緯を追いにくい
- 対応中なのか、確認済みなのか、完了したのかが分かりにくい

Rebioは、こうした現場の情報共有を少しでも分かりやすくするために開発したデモアプリです。

## 設計思想：普段は迷わず、必要な時だけ詳しく確認できる

Rebioは、多機能さよりも「日常業務の中で迷わず使えること」を重視しています。

| 原則 | 実装 |
|---|---|
| **入力はシンプルにする** | テキストの自由入力を基本とし、クイック入力テンプレートで補助 |
| **必要な情報だけ補足する** | 担当者未設定・報告が必要そうな内容などにだけヒントを表示 |
| **候補生成は補助にとどめる** | 自動整理・タスク候補生成・タグ付けは、最終判断の前段階として扱う |
| **重要なものを見つけやすくする** | 「対応が必要」な申し送りを分かりやすく表示し、共有事項は軽量に表示 |
| **すべてをタスク化しない** | 対応が必要な申し送りだけをタスク化し、共有事項や通常記録は無理にタスクにしない |
| **画面構成を分かりやすくする** | ホーム・記録・申し送り・書類を中心に、主要な操作へすぐ移動できるようにする |
| **詳細操作は必要な時に表示する** | フィルター・ステータス変更・修正などは、画面を広げて確認できるようにする |
| **書類と業務をゆるくつなげる** | 書類は申し送り・タスクと紐づけられるが、単なるファイル置き場としても使える |
| **現場ごとの使い方に合わせる** | カテゴリ・タグ・確認レベルは自由に設定できるようにする |

現場の担当者が「次に何を確認すればいいか」「誰が対応するのか」を把握しやすい画面を目指しています。

## Rebioで意識したこと

Rebioでは、チャット・タスク管理・ファイル管理の要素を取り入れていますが、中心に置いているのは「業務の経緯を追えること」です。

- **申し送りの対象を選べる**  
  全体だけでなく、部署・個人・役職・シフトに向けて共有できます。対象を選ぶUIは軽量にし、複雑な組織管理機能は現時点では持たせていません。

- **確認状況を把握できる**  
  複数人に共有した場合、「既読 2/4」「確認 1/3」のように、どこまで確認されたかを把握できます。

- **コメントは補足記録として扱う**  
  コメント機能は、チャットのような会話を広げるためではなく、「確認しました」「明日対応します」など、申し送りに対する補足を残すためのものとして設計しています。

- **情報と行動をつなげる**  
  申し送りの中で対応が必要なものはタスクとして扱い、関連書類や過去のコメントと一緒に確認できるようにしています。

## デザイン方針

Rebioは、現場の人が落ち着いて使えるように、強い色や過剰な装飾を避け、必要な情報が自然に目に入るUIを目指しています。

- **落ち着いたカラーパレット** — ブランドカラーはブルー・ティール系を中心にし、警告色は必要な場面に限定
- **白カードと薄いグレー背景** — 情報のまとまりが分かるように、カードと境界線で画面を整理
- **重要度に応じた表示** — 共有事項は軽く、対応が必要なものは少し目立つように表示
- **統一感のあるUI** — バッジ・カード・ボタンの形や余白をそろえ、業務アプリとして見やすくする

## 主な機能

### 1. 勤務ログ入力（`/logs/new`）
- 勤務種別・本文・次の担当者・重要度を記録
- 業種別サンプルログ・クイック入力テンプレートで操作を体験できる
- 入力後、Rebioが申し送り候補を整理

### 2. 整理候補の確認画面（`/handover/review`）
- 処理フロー（ログ解析→候補抽出→重要度判定→確認必須判定→本人の確認）を表示
- 「共有事項」「確認してほしい」「対応が必要」の3種類に分類
- 重要度・信頼度・元ログの対応箇所を表示
- 「確認済み」「修正する」「不要」の3択で一件ずつ判断

### 3. 申し送りを確認（`/handover`）
- 申し送りを種類ごとに表示
  - **共有事項** … 対応不要の情報共有
  - **確認してほしい** … 既読・確認状態を管理
  - **対応が必要** … タスク化・期限・担当者・ステータスを表示
- カテゴリ・重要度・ステータス・担当者でフィルタリング、キーワード検索
- 送信対象指定：全体・部署・個人・役職・シフトの5種類から共有先を選択
- 既読・確認の集計：「既読 2/4」「確認 1/3」のように確認状況を表示
- 補足コメント：「確認しました」「明日対応します」などの経過を残せる

### 4. 次の行動（`/tasks`）
- 「対応が必要」と判断された申し送りだけをタスク化
- 期限・担当者・完了管理
- 期限超過を視覚的に表示

### 5. 勤務ログ一覧（`/logs`）
- 過去のログを勤務種別でフィルタリング
- 生成された申し送り件数を表示

### 6. 書類（`/library`・`/library/[id]`）
- 左：カテゴリ一覧／中央：ファイル一覧／右：選択中ファイルの簡易プレビュー、というエクスプローラー型UI
- PDF・Excel・Word・PowerPoint・CSV・画像のプレビュー（モック）
- カテゴリ・タグを自由設定
- ピン留め・最近開いた書類、申し送り・タスクとの紐づけ
- リスト表示／コンパクト表示の切り替え

### 7. ダッシュボード（`/`）・デモページ（`/demo`）
- 対応が必要な件数・緊急件数・未完了タスク・本日ログ数のサマリー
- デモ操作ガイド付きの導線カード
- デモページでRebioの概要・設計意図・画面案内を確認可能

## 候補生成ロジックを使う上での安全設計

Rebioでは、システムが作成した候補をそのまま確定するのではなく、必ず人が確認してから登録する設計にしています。

| 安全設計 | 実装内容 |
|---|---|
| **システムは候補を提示するだけ** | 生成された申し送りは「整理候補」として扱い、登録前に本人が確認 |
| **確認必須フラグ** | 対応が必要な項目や、重要度が高い項目は確認なしでは登録できない |
| **元ログの根拠表示** | 全候補に「元ログのどの部分から抽出したか」を表示 |
| **確認時間の警告** | ページ開封から30秒未満で登録しようとすると警告モーダルを表示 |
| **断定的な表現を避ける** | 「判断しました」ではなく「候補を作成しました」「確認してください」に統一 |
| **処理フローを見せる** | 整理のステップ（解析→抽出→判定→確認）を画面上部に表示 |

現在のデモでは、候補生成はルールベースのキーワードマッチングで実装しています。将来的にはLLMとの接続によって、より柔軟な候補整理に拡張する想定です。

## 書類機能の設計意図

Rebioの書類機能は、業務で使う資料を申し送りやタスクと一緒に確認しやすくするためのものです。

- **用途を限定しすぎない** — 点検表・マニュアル・報告書など、現場ごとに自由に使えるようにする
- **必要な時だけ紐づける** — 書類は申し送り・タスクと紐づけできるが、単なるファイル置き場としても使える
- **一覧性を重視する** — ファイル一覧と簡易プレビューを同じ画面で確認できるようにする
- **複数形式を想定する** — PDF・Excel・Word・PowerPoint・CSV・画像に対応したデモ用プレビューを用意
- **アップロードは段階的に拡張する** — 公開デモのlocalStorageモードでは実ファイルアップロードは行わず、Supabaseモードでのアップロード対応を別途用意している

## 起動手順

### localStorageデモモードで起動する（デフォルト）

```bash
# 依存関係をインストール
npm install

# 開発サーバーを起動
npm run dev

# ブラウザでアクセス
open http://localhost:3000
```

環境変数を何も設定しなくても、自動的にlocalStorageデモモードで動作します。

### Supabaseモードで起動する

```bash
# 1. .env.local.example を .env.local にコピー
cp .env.local.example .env.local

# 2. .env.local を編集し、Supabaseの接続情報を設定
# NEXT_PUBLIC_REBIO_DATA_MODE=supabase
# NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
# NEXT_PUBLIC_SUPABASE_ANON_KEY=xxxxx

# 3. Supabase側でテーブルとStorage bucketを作成
#    docs/supabase-schema.sql を参考にテーブルを作成
#    Storage > New bucket で "documents" という名前のバケットを作成

# 4. 開発サーバーを起動
npm run dev
```

接続情報が正しくない場合や未設定の場合は、自動的にlocalStorageデモモードにフォールバックします。

### ビルド確認

```bash
npm run build
```

### Vercelへのデプロイ

1. GitHubリポジトリと連携し、Vercelで新規プロジェクトを作成
2. Framework Presetは自動的に「Next.js」が選択される
3. Supabaseモードで運用する場合は、Vercelの Project Settings > Environment Variables に以下を設定
   - `NEXT_PUBLIC_REBIO_DATA_MODE` = `supabase`
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
4. localStorageデモモードのまま公開する場合は環境変数の設定は不要

## 権限の見せ方

本格的な権限管理画面はまだ実装していませんが、将来的に以下の4段階を想定しています。

| 権限 | 想定できること |
|---|---|
| **管理者** | チーム全体の申し送り・タスク・書類の閲覧と管理、部署・役職・シフトの設定、メンバーの追加・権限変更 |

現時点ではすべての操作が一般メンバー相当の権限で行える単一ユーザーのデモです。`docs/supabase-rls.md` に、実装時のアクセス制御方針を整理しています。

## デモデータについて

初回アクセス時に以下のサンプルデータが自動投入されます。

- **勤務ログ** 8件（介護・警備・小売・物流の4業種）
- **申し送り** 12件（カテゴリ・重要度・信頼度付き）
- **タスク** 6件（対応中・未着手・完了を含む）
- **書類** 9件（PDF・Excel・Word・PowerPoint・CSV・画像）

**すべて架空のデータです。** 実在する個人・企業・施設のデータは一切含まれていません。

データはlocalStorageに保存されるため、ブラウザを変えると初期化されます。  
データをリセットしたい場合は、ブラウザのDevTools > Application > LocalStorageからクリアしてください。

## Supabase移行設計と実装状況（v12〜v13）

現在はlocalStorageベースのデモMVPですが、Supabaseへの移行設計と、切り替え可能な実装の土台を追加しています。

### 設計（v12）
- **データベース設計**：`docs/supabase-schema.sql` にテーブル定義を用意（profiles・teams・handovers・tasks・documentsなど15テーブル）
- **RLS方針**：`docs/supabase-rls.md` に、チーム境界・送信対象に基づくアクセス制御方針を整理
- **認証方針**：Supabase Authを使い、メールログインまたはGoogleログインを想定。ログイン後に `profiles` を作成し、`team_members` で所属チームを管理する
- **移行ロードマップ**：`docs/production-roadmap.md` に、v13（データ永続化）・v14（ファイルストレージ）・v15（運用仕上げ）の計画を記載

### データモードの切り替え（v13）

Rebioは環境変数 `NEXT_PUBLIC_REBIO_DATA_MODE` で動作モードを切り替えられます。

| 値 | 説明 |
|---|---|
| `local`（デフォルト） | localStorageのみで動作するデモモード。環境変数が未設定の場合も自動的にこちらになる |
| `supabase` | Supabaseに接続して動作するモード。`NEXT_PUBLIC_SUPABASE_URL` と `NEXT_PUBLIC_SUPABASE_ANON_KEY` の設定が必要 |

`supabase` を指定していても接続情報が欠けている場合は、自動的に `local` モードにフォールバックします。アプリがクラッシュすることはありません。

### 環境変数

`.env.local.example` をコピーして `.env.local` を作成してください。

```env
NEXT_PUBLIC_REBIO_DATA_MODE=local
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
```

### repository層

`app/lib/repositories/` に、データ取得・保存をモードに応じて切り替えるrepository層を新設しました（`handovers.ts` / `workLogs.ts` / `tasks.ts`）。v13時点ではSupabaseへの読み書きロジックは一部未実装で、Supabaseモードでも実際のデータ操作はlocalStorageに委譲されます（次フェーズで本実装予定）。既存ページは引き続き `app/lib/store.ts` を直接利用しており、動作に影響はありません。

### ログイン画面

`/login` ページを追加しました。localモードではログイン不要で「デモユーザーとして続ける」ボタンからそのまま利用できます。Supabaseモードではメールアドレス入力フォームを表示します（実際の送信処理は次フェーズで実装）。

**この構成はまだ実運用を保証するものではありません。** 将来的に認証・チーム管理・クラウド保存・ファイル管理へ拡張できる構成を明文化し、土台を作った段階です。

## 実ファイルアップロード（v14）

Supabaseモードでは、`/library` から実際のファイルをアップロードできます。

- **Storage bucket**：`documents`
- **対応ファイル**：PDF・PNG・JPG・JPEG・WEBP・CSV（上限10MB）
- **保存内容**：Supabase Storageに実ファイル本体を保存し、`documents` テーブルにメタデータ（ファイル名・カテゴリ・タグ・説明・サイズ・アップロード者）を保存
- **localStorageデモモードでは実アップロードはできません**。アップロードを試みると「デモモードのため利用できません」という案内が表示されます

### エラー処理

以下の状況では、原因に応じた案内を表示します。

- Supabase未設定（localモード）でアップロードを試みた場合
- 非対応のファイル形式を選んだ場合
- ファイルサイズが10MBを超える場合
- Storage bucketが未作成の場合
- 通信エラーが発生した場合

### 削除確認

書類の削除は、Supabaseモード・localモードいずれでも確認ダイアログを表示してから実行します。誤操作による削除を防ぐため、確認なしでの削除はできません。

---

## 実運用データを扱わないことについて

**本アプリはポートフォリオ・デモ用途として開発されています。**

実運用に向けて以下の追加実装が必要です：

- 認証・認可（Auth.js・Supabase Auth 等）
- サーバーサイドのデータ永続化（PostgreSQL・Supabase 等）
- 監査ログ・データ暗号化
- 個人情報保護方針・利用規約の整備
