# Rebio Supabase RLS（Row Level Security）方針

このドキュメントは、`docs/supabase-schema.sql` の各テーブルに対する RLS ポリシーの方針をまとめたものです。
実際のポリシーSQLは v13 の実装時に `supabase/migrations/` に追加する想定です。

## 基本原則

1. **チーム境界を越えたデータアクセスは常に禁止する**。すべてのテーブルは `team_id`（または申し送り経由での間接的な team_id）でスコープされる
2. **申し送りの送信対象に含まれるユーザーのみが閲覧できる**。全体向け（`all`）は同チーム全員が対象になる
3. **管理者（`team_members.permission = 'admin'`）は同チーム内のデータを広く閲覧・管理できる**
4. **一般メンバーは自分に関係するもの（自分宛て・自分の部署/役職/シフト宛て・全体向け）のみ閲覧・操作できる**
5. 過度に複雑な権限分岐は避け、「同チームかどうか」と「対象に含まれるかどうか」の2段階で判定する

## テーブルごとの方針

### `profiles`
- 本人は自分の行を読み書きできる
- 同チームのメンバーは他メンバーのプロフィール（名前程度）を閲覧できる

### `teams` / `team_members`
- チームメンバーは自分の所属チーム情報を閲覧できる
- チームへの新規メンバー追加・権限変更は `permission = 'admin'` のユーザーのみ可能

### `departments` / `roles` / `shifts`
- 同チームのメンバーは閲覧可能
- 作成・編集・削除は管理者のみ

### `work_logs`
- 作成は同チームのメンバーなら誰でも可能
- 閲覧は同チームのメンバーに限定（勤務ログ自体は機密性が高くないため、同チーム内は広く共有する）

### `handovers`
- **作成**：同チームのメンバーであれば作成可能
- **閲覧**：以下のいずれかを満たす場合のみ
  - 送信対象（`handover_audiences`）に自分が含まれる（個人・部署・役職・シフト・全体のいずれかで一致）
  - 自分が同チームの管理者である
  - 自分がその申し送りの作成者（`author_id`）である
- **更新（ステータス変更等）**：閲覧可能なユーザーであれば操作可能。ただし「対応が必要」なものの完了操作は、対象者または管理者に限定することを検討する（初期実装では閲覧可能=操作可能で問わない）

### `handover_audiences`
- 閲覧は対応する `handovers` を閲覧できるユーザーに従う（`handovers` のRLSに委譲）
- 作成は申し送り作成時のみ（handoversの作成権限に従う）

### `read_receipts`
- **自分の行のみ更新可能**（`read_at` / `confirmed_at` を自分のuser_idの行に対してのみ書き込める）
- 閲覧は、対応する `handovers` を閲覧できるユーザーであれば全対象者の既読状況を閲覧できる（集計表示のため）
- 個々の未読者名を晒す設計を避ける方針は維持するため、UI側で集計表示に留める（RLS上は閲覧できるがUIで抑制する）

### `handover_comments`
- **作成**：対応する `handovers` を閲覧できるユーザーのみコメント可能
- **閲覧**：対応する `handovers` を閲覧できるユーザー
- 編集・削除は当面実装しない（v11の方針を維持）

### `tasks`
- 閲覧・操作は対応する `handovers` を閲覧できるユーザーに従う
- タスクは「対応が必要」な申し送りからのみ生成されるため、handoversのRLSと整合する

### `documents` / `document_links`
- 閲覧は同チームのメンバー全員（書類は「用途を決めすぎない」設計思想のため、対象者限定にはしない）
- アップロード・削除は同チームのメンバーであれば可能（初期実装ではシンプルに。将来的にアップロード者のみ削除可、のように厳格化する余地はある）

### `activity_logs`
- 閲覧は同チームのメンバー
- 作成はシステム（アプリケーションロジック）から自動的に行われる。ユーザーが直接書き込むことは想定しない

## 実装方針（v13向けメモ）

- RLSは `auth.uid()` と `team_members` を JOIN する形で判定する
- 「送信対象に自分が含まれるか」の判定は、`handover_audiences` の `audience_type` ごとに以下のように分岐する
  - `all` → 同チームなら常に真
  - `user` → `target_user_id = auth.uid()`
  - `department` / `role` / `shift` → `team_members` の該当カラムと一致するか
- パフォーマンスを考慮し、上記の判定はSQL関数（`is_handover_visible_to(handover_id, user_id)` のようなヘルパー関数）として定義し、各テーブルのポリシーから呼び出す形にすることを推奨する
